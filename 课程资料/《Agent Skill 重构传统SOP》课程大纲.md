# 《Agent Skill 重构传统SOP》课程大纲

## 课程基本信息

**课程名称**：Agent Skill 重构传统SOP  
**课程时长**：1小时20分钟  
**课程定位**：实战应用课程  
**适用人群**：
- 产品经理、业务分析师：希望用AI提升工作流程效率
- 技术团队负责人：需要建立团队级AI工作流标准
- AI应用开发者：希望系统化管理AI提示词和工作流

**前置要求**：
- 了解基本的AI对话交互方式

- 无需编程基础

## 课程开设背景

在AI应用的早期阶段，我们习惯于通过"小作文"式的长提示词（Prompt）来引导AI。然而，随着业务复杂度的提升，这种方式遇到了三大瓶颈：

### 痛点1：维护成本高
- 每次对话都要重新描述需求和标准
- 长提示词难以复用和版本管理
- 团队成员的提示词五花八门，质量参差不齐

### 痛点2：上下文拥挤
- Context Window虽然在变大，但拥挤的上下文导致AI注意力分散
- 关键信息"Lost in the Middle"（迷失在中间）
- Token消耗大，成本高

### 痛点3：知识隔离
- 专用Agent之间能力难以互通
- 跨领域任务需要重新构建整套流程
- 企业知识难以沉淀和传承

**Agent Skills 的出现标志着AI应用进入了标准化、模块化的新阶段。**

---

## 课程目标

通过本课程学习，你将能够：

1. **理解范式转移**：从"小作文式Prompt"到"可复用的Skill"
2. **掌握改造方法**：将传统SOP流程改造为Agent Skill
3. **独立实战**：使用skill-creator工具创建自己的专属Skill
4. **建立体系**：为团队建立标准化的AI工作流库

**预期成果**：
- 课后能够将1个现有工作流改造为Skill
- 将工作效率提升50%以上（减少重复性沟通）
- 建立个人或团队的Skill库

## 课程内容结构

### **开场：从一个真实场景说起**（5分钟）
- 一个营销经理的烦恼：每周分析营销数据的痛苦
- 传统方式 vs Agent Skill方式的对比
- 引出课程核心议题

---

### **第一部分：传统SOP的困境**（15分钟）

#### 1.1 案例：营销活动数据分析SOP（8分钟）
- 传统分析流程的7个步骤
- 每个步骤需要的详细提示词
- 多轮对话的低效和易错

#### 1.2 "小作文式Prompt"的三大问题（7分钟）
- **重复劳动**：每周重复相同的长提示词
- **质量不稳定**：不同人写的Prompt效果差异大
- **知识无法沉淀**：流程变更时需要全员重新培训

---

### **第二部分：Agent Skill的革命性突破**（15分钟）

#### 2.1 什么是Agent Skill？（5分钟）
- 从"长文本指令"到"文件系统标准"
- Skills的核心理念：通用Agent + 可插拔Skill
- Skills的三大特征：可移植、可组合、渐进式加载

#### 2.2 改造后的效果演示（10分钟）
- **Before**：三段长提示词，多轮对话
- **After**：一句话触发完整流程
- 对比数据：
  - 提示词长度：从1500字 → 20字
  - 交互轮次：从3轮 → 1轮
  - 时间消耗：从15分钟 → 3分钟
- 实际演示：使用analyzing-marketing-campaign Skill

---

### **第三部分：环境搭建与配置**（15分钟）

#### 3.1 Claude Code安装（5分钟）
- 下载与安装步骤
- 基础界面介绍
- 验证安装成功

#### 3.2 Skill的安装与管理（10分钟）
- Skill的三种安装位置：
  - 项目级：`.claude/skills/`（团队共享）
  - 全局级：`~/.claude/skills/`（个人使用）
  - 临时导入（演示测试）
- 安装第一个Skill：analyzing-marketing-campaign
- 验证Skill加载成功
- 启用/禁用Skill的方法

---

### **第四部分：深度解析Skill改造流程**（20分钟）

#### 4.1 Skill的文件结构（5分钟）
```
analyzing-marketing-campaign/
├── SKILL.md           # 核心文件
├── references/        # 参考文档
│   └── budget_reallocation_rules.md
├── scripts/          # 可执行脚本（可选）
└── assets/           # 模板和资源（可选）
```

#### 4.2 SKILL.md的解剖（10分钟）
##### 第一层：YAML Frontmatter（元数据）
```yaml
---
name: analyzing-marketing-campaign-zh
description: 分析每周多渠道营销活动的绩效数据...
---
```
- name：命名规则（kebab-case）
- description：触发关键词设计技巧

##### 第二层：Markdown正文（主指令）
- 输入要求：明确数据格式
- 数据质量检查：标准化流程
- 漏斗分析：计算公式和基准
- 效率分析：ROAS、CPA、净利润
- 输出格式：标准化表格和解读

##### 第三层：References（按需加载）
- 预算重新分配规则
- 复杂决策矩阵
- 只在需要时才加载到上下文

#### 4.3 传统SOP到Skill的改造步骤（5分钟）
1. **识别工作流**：提取核心步骤
2. **标准化输入输出**：定义数据格式
3. **提炼判断逻辑**：if-else规则
4. **分层组织内容**：元数据-主指令-参考
5. **设计触发机制**：description优化
6. **测试与迭代**：验证触发准确性

---

### **第五部分：实战练习 - 使用skill-creator创建Skill**（20分钟）

#### 5.1 需求澄清Skill的场景（3分钟）
- 问题：AI经常对模糊需求做出错误假设
- 目标：创建一个自动提问澄清的Skill
- 触发条件：需求有多种解释或关键细节不清楚

#### 5.2 使用skill-creator创建流程（12分钟）
##### Step 1：启动skill-creator（2分钟）
```
帮我使用 skill-creator 创建一个新的 Skill
```

##### Step 2：回答引导问题（8分钟）
- **Q1: Skill名称？**
  ```
  clarify-demand-uncertainty
  ```

- **Q2: 功能描述？**
  ```
  在实施之前澄清需求。当出现严重疑问时使用。
  适用于请求有多种合理解释或关键细节不清楚的场景。
  ```

- **Q3: 核心用例？**（提供2-3个具体场景）
  ```
  用例1：前端功能开发
  用户说："在设置页面加一个夜间模式开关"
  Skill应该提问：开关位置、持久化方式、生效范围等
  
  用例2：后端API开发
  用户说："写一个用户注册接口"
  Skill应该提问：必填字段、验证方式、密码规则等
  ```

- **Q4: 工作流程？**
  ```
  1. 判断请求是否不明确（检查目标、范围、约束等）
  2. 提出1-5个关键澄清问题（提供多选项、建议默认值）
  3. 暂停等待用户回答（不执行任何实施操作）
  4. 确认理解后重述需求并继续
  ```

##### Step 3：审查与优化（2分钟）
- 检查生成的SKILL.md
- 调整description提高触发准确率
- 完善"何时使用"和"何时不使用"

#### 5.3 测试新创建的Skill（5分钟）
##### 触发测试
```
测试1: "在首页加一个搜索框"（应触发）
测试2: "在Header.tsx第25行添加一个红色按钮"（不应触发）
```

##### 工作流测试
- 验证是否提问
- 检查问题质量
- 确认暂停等待机制

---

### **第六部分：最佳实践与进阶技巧**（8分钟）

#### 6.1 设计高质量Skill的五大原则（4分钟）
1. **精准触发**：description包含明确的关键词
2. **结构清晰**：使用编号、表格、项目符号
3. **渐进加载**：合理使用references目录
4. **可组合性**：考虑与其他Skill的配合
5. **持续迭代**：根据使用反馈优化

#### 6.2 团队级Skill管理（4分钟）
- 项目级Skill的版本控制（Git）
- Skill库的组织结构
- 团队Skill的命名规范
