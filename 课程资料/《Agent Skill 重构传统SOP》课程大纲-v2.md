# 《Agent Skill 重构传统SOP》课程大纲

## 课程基本信息

**课程名称**：Agent Skill 重构传统SOP  
**课程时长**：1小时20分钟  
**课程定位**：实战应用课程  

**适用人群**：
- 产品经理、业务分析师：希望用AI提升工作流程效率
- 技术团队负责人：需要建立团队级AI工作流标准
- 研发团队：希望用AI优化重复性工作流程

**前置要求**：
- 了解基本的AI对话交互方式
- 有一定的业务流程思维（了解SOP概念）
- 无需编程基础

**课程特色**：
- 对比鲜明：直观展示传统方法 vs Skill方法的效率差异
- 实战导向：两个完整案例的动手实操
- 即学即用：课后可立即应用到实际工作场景

### 1.2 案例一：营销数据分析的效果对比【10分钟】

> **侧重点**：展示Skill方法的便捷省力

#### Before：传统"小作文"式Prompt（5分钟）

**演示内容**：
- **第一轮对话**（300字）：上传数据 + 说明列名 + 数据质量检查要求 + 漏斗分析公式 + 基准对比
- **第二轮对话**（250字）：补充效率指标计算（ROAS、CPA、净利润）+ 成本参数 + 目标对比
- **第三轮对话**（100字）：上传预算调整规则文档 + 说明约束条件

**数据统计**：
```
📝 提示词总长度：650字（三段累计）
🔄 交互轮次：3轮
⏱️ 总耗时：15分钟
💰 Token消耗：约5000 tokens
🎯 易错点：5-8个（公式错误、参数遗漏等）
```

**用户痛点**：
- 😓 每周重复相同操作
- 🤔 需要记住复杂的参数和公式
- 😰 担心遗漏关键步骤
- 📋 规则文档需要反复上传

---

#### After：使用Skill的方式（5分钟）

**演示内容**：
```
我上传了本周的营销活动数据。请分析数据，帮助我了解各渠道的表现情况
（漏斗绩效和效率绩效），以及是否需要重新分配预算。

我可以在渠道间调配最多 ¥10,000 的预算，单个渠道的预算增幅不超过 15%。
```

**数据统计**：
```
📝 提示词总长度：80字
🔄 交互轮次：1轮
⏱️ 总耗时：3分钟
💰 Token消耗：约1500 tokens
🎯 易错点：0个（所有规则已内置）
```

**用户体验**：
- 😊 像和专家对话一样简单
- ✅ 无需记住任何细节
- 🚀 专注于业务决策而非操作流程
- 🔄 所有规则自动应用

---

**效率对比总结**：

| 维度 | 传统Prompt | Skill方式 | 提升幅度 |
|------|-----------|----------|---------|
| 文字输入 | 650字 | 80字 | **减少88%** |
| 交互轮次 | 3轮 | 1轮 | **减少67%** |
| 时间消耗 | 15分钟 | 3分钟 | **节省80%** |
| Token消耗 | 5000 | 1500 | **节省70%** |
| 人为错误 | 5-8个 | 0个 | **减少100%** |

**现场演示**：
- 实际操作展示Skill的触发和执行过程
- 展示AI输出的专业性和完整性

### 1.3 案例二：需求澄清的效果对比【7分钟】

> **侧重点**：展示对团队/重复工作的直观提效

#### Before：传统方式的问题（3分钟）

**场景重现**：
```
用户："在设置页面加一个夜间模式开关"

AI直接开始写代码：
✅ 创建了Toggle组件
✅ 添加了状态管理
❌ 但不知道放在哪个位置
❌ 不知道是否要持久化
❌ 不知道生效范围（全局 vs 当前页）

结果：返工！需要重新澄清需求，再次修改代码。
```

**团队痛点**：
- 😓 每个研发都要手动追问细节
- 🔄 追问的问题五花八门，不标准
- ⏱️ 来回确认浪费大量时间
- 💸 返工成本高（2-3倍工作量）

**数据统计**：
```
🔄 平均澄清轮次：3-4轮
⏱️ 平均耗时：10-15分钟
🎯 返工率：40%（需求理解偏差）
```

---

#### After：使用Skill的方式（4分钟）

**演示内容**：
```
用户："在设置页面加一个夜间模式开关"

AI启动需求澄清Skill：
"在我开始之前，我需要澄清几个问题：

1. 开关位置？
   a) 设置页面顶部（推荐）
   b) 设置页面底部
   c) 其他位置：_______

2. 是否持久化？
   a) 是，保存到localStorage（推荐）
   b) 否，仅会话期间有效

3. 生效范围？
   a) 全局（所有页面）- 推荐
   b) 仅当前页面

回复：defaults（或 1a 2a 3a）"
```

**Skill的价值**：
- 🎯 标准化提问：每次都问对问题
- ⚡ 快速响应：提供多选项和默认值
- ✅ 一次做对：避免返工
- 📋 团队统一：所有人使用相同的澄清流程

**效率对比总结**：

| 维度 | 传统方式 | Skill方式 | 提升幅度 |
|------|---------|----------|---------|
| 澄清轮次 | 3-4轮 | 1轮 | **减少75%** |
| 澄清时间 | 10-15分钟 | 2-3分钟 | **节省80%** |
| 返工率 | 40% | 5% | **降低88%** |
| 团队一致性 | 低（每人问法不同） | 高（统一标准） | **质的提升** |

**团队价值**：
- 对于10人团队，每周可节省 **10小时** 的需求澄清时间

### 2.2 案例一实操：营销数据分析Skill【25分钟】

> **教学策略**：边拆解、边理解、边实践

#### A. 识别工作流（5分钟）

**演示文件**：`prompts_initial_conversation.md`

**工作流提取**：
```
步骤1：数据质量检查
  ├─ 检查缺失值
  ├─ 验证数值合理性
  └─ 识别异常值

步骤2：漏斗分析
  ├─ 计算CTR（点击率）
  ├─ 计算CVR（转化率）
  └─ 与基准对比

步骤3：效率分析
  ├─ 计算ROAS（广告回报率）
  ├─ 计算CPA（获客成本）
  └─ 计算净利润

步骤4：预算调整建议（可选）
  ├─ 读取预算调整规则
  └─ 应用决策矩阵
```

**教学要点**：
- 如何从传统SOP中提取核心步骤
- 识别输入（数据文件）和输出（分析报告）
- 标记判断逻辑（if-else分支）

---

#### B. 标准化输入输出（5分钟）

**输入标准化**：
```yaml
输入格式：CSV文件
必需列：
  - 日期 (Date)
  - 渠道 (Channel)
  - 展示次数 (Impressions)
  - 点击次数 (Clicks)
  - 转化次数 (Conversions)
  - 支出 (Spend)
  - 收入 (Revenue)
  - 订单数 (Orders)
```

**输出标准化**：
```yaml
输出格式：
  1. 数据质量检查报告（文本）
  2. 漏斗分析表（表格）
  3. 效率分析表（表格）
  4. 渠道绩效总结（文本）
  5. 预算调整建议（表格，可选）
```

**教学要点**：
- 为什么要标准化？（确保一致性、可复用）
- 如何设计输入格式？（考虑常见场景）
- 如何设计输出格式？（清晰、可操作）

---

#### C. 提炼判断逻辑（5分钟）

**示例1：渠道差异化处理**
```
IF 渠道 == "小程序推送":
    CTR = "N/A"（无展示次数）
    CVR = 正常计算
ELSE:
    CTR = 点击次数 / 展示次数
    CVR = 转化次数 / 点击次数
```

**示例2：状态判断**
```
IF ROAS >= 目标ROAS:
    状态 = "✓ 达标"
ELSE:
    状态 = "✗ 未达标"
```

**示例3：预算调整规则**
```
IF 净利润 < 0 AND ROAS < 50%:
    分类 = "大幅削减"
    调整幅度 = -45%
ELSE IF ROAS > 目标 AND CPA < 目标:
    分类 = "增加投入"
    调整幅度 = +20%
...
```

**教学要点**：
- 如何从业务规则中提炼if-else逻辑
- 如何处理边界条件和特殊情况
- 如何设计决策矩阵

---

#### D. 创建SKILL.md文件（8分钟）

**Step 1：编写YAML Frontmatter（2分钟）**

```yaml
---
name: analyzing-marketing-campaign
description: 分析每周多渠道营销活动的绩效数据。用于分析多渠道数字营销数据，计算漏斗指标（CTR、CVR）并与基准对比，计算成本和收益效率指标（ROAS、CPA、净利润），或根据绩效规则获取预算重新分配建议。
---
```

**教学要点**：
- name必须与文件夹名称一致
- description要包含触发关键词："营销"、"绩效"、"CTR"、"预算"
- description要说明"做什么"和"何时用"

---

**Step 2：编写主指令（4分钟）**

```markdown
# 营销活动分析

## 输入要求
期望以 CSV 格式提供营销活动数据，包含以下列：
- 日期、渠道、展示次数、点击次数...

## 数据质量检查
1. 检查缺失值和空单元格
2. 验证数值列中没有负值
...

## 漏斗分析
按渠道计算：
- CTR = 点击次数 / 展示次数 × 100
- CVR = 转化次数 / 点击次数 × 100
...

## 效率分析
按渠道计算：
- ROAS = 收入 / 支出
- CPA = 支出 / 转化次数
...

## 预算重新分配
如果用户询问预算重新分配，请阅读 
`references/budget_reallocation_rules.md`...
```

**教学要点**：
- 使用清晰的层次结构（##、###）
- 使用列表和表格展示信息
- 保持简洁，复杂规则拆分到references/

### 2.3 过渡：什么是Skill？【5分钟】

> **教学策略**：从案例中总结提炼概念

**通过案例反思**：
```
刚才我们做了什么？
1. 把传统的长Prompt → 转化为结构化的文件夹
2. 把一次性的指令 → 转化为可复用的技能
3. 把散落的规则 → 转化为按需加载的知识库
```

---

**Skill的三层架构**：

| 层级 | 内容 | 何时加载 | Token消耗 |
|-----|------|---------|----------|
| **Layer 1** | YAML Frontmatter | 系统启动时（始终） | 极低（~50 tokens） |
| **Layer 2** | SKILL.md正文 | 触发时 | 中等（~2000 tokens） |
| **Layer 3** | references/ | 需要时 | 按需（~1000-5000 tokens） |

**工作流程**：
```
1. Agent启动 → 加载所有Skill的元数据（Layer 1）
   ↓ "我知道有100个技能可用"
   
2. 用户请求 → 匹配相关Skill → 加载主指令（Layer 2）
   ↓ "这个任务需要营销分析Skill"
   
3. 执行过程 → 需要复杂规则 → 加载参考文档（Layer 3）
   ↓ "预算调整需要详细规则，现在加载"
```

---

**Skill的核心价值**：

1. **可复用性**
   - 一次创建，永久使用
   - 告别重复的"小作文"

2. **标准化**
   - 团队统一流程
   - 避免人为差异

3. **渐进式加载**
   - 最小化Token消耗
   - 保护Context Window

4. **可组合性**
   - 多个Skill可以组合使用
   - 构建复杂工作流

5. **可移植性**
   - 在Claude.ai、Claude Code、API间无缝使用
   - 开放标准，生态共享

---

**skill-creator响应**：

我将引导你创建一个新的Skill。让我们逐步定义它：

```

---

#### B. 回答引导问题（6分钟）

**Q1: Skill名称**
```
输入：clarify-demand-uncertainty

✅ kebab-case格式
✅ 动词+名词形式
✅ 语义清晰
```

---

**Q2: Skill描述**
```
输入：在实施之前澄清需求。当出现严重疑问时使用。
适用于请求有多种合理解释或关键细节（目标、范围、

**Q3: 核心用例（2-3个）**
```
输入：

用例1：前端功能开发
用户说："在设置页面加一个夜间模式开关"
Skill应该提问：开关位置、持久化方式、生效范围、

用例2：后端API开发

用例3：代码重构

Skill应该提问：性能目标、约束条件、是否可以

```

**教学要点**：
- 用例要具体，不要泛泛而谈
- 用例要覆盖主要场景
- 用例要说明"输入"和"期望的Skill行为"

---

**Q4: 工作流程步骤**
```
输入：

1. 判断请求是否不明确
   - 检查：目标是否清晰？
   - 检查：完成标准是否明确？
   - 检查：范围是否定义？
   - 检查：约束是否清楚？
   - 检查：环境是否识别？

2. 提出1-5个关键澄清问题
   - 优化可扫描性（编号、简短）
   - 提供多项选择
   - 建议合理默认值
   - 包含快速响应路径（如"defaults"）

3. 暂停等待用户回答
   - 不运行命令
   - 不编辑文件
   - 不制定详细计划

4. 确认理解后重述需求并继续
   - 用1-3句话重述需求
   - 明确假设
   - 开始实施
```

**教学要点**：
- 工作流要步骤清晰
- 每步要可执行
- 要包含验证门（不做什么）
- 要有明确的暂停点

---

**Q5: 额外配置**
```

A: 不需要，仅依赖Claude的内置能力
```

## 第三部分：总结与延伸【10分钟】

> **目标**：巩固认知、提炼方法论、指引行动

### 3.1 再次强调：什么是Skill？【3分钟】

**从两个案例回顾**：

```
案例一（营销分析）：
我们把一个需要3轮对话、650字提示词的复杂分析流程，
转化为一个自动触发、标准化执行的Skill。

案例二（需求澄清）：
我们用skill-creator在10分钟内创建了一个能够标准化
团队提问流程、减少80%返工的Skill。
```

---

**Skill的本质**：

1. **技能化的知识封装**
   - 不再是"临时的指令"，而是"持久的技能"
   - AI从"听指令的助手"变成"拥有专业技能的专家"

2. **文件系统标准**
   - 基于文件夹和Markdown的开放标准
   - 可版本控制、可团队共享、可生态协作

3. **渐进式披露机制**
   - 三层架构：元数据 → 主指令 → 参考资料
   - 按需加载，最小化Token消耗

4. **可组合的能力单元**
   - 每个Skill专注一个领域
   - 多个Skill组合完成复杂任务

---

### 3.2 Skill的价值与适用场景【4分钟】

**核心价值总结**：

| 价值维度 | 个人价值 | 团队价值 | 企业价值 |
|---------|---------|---------|---------|
| **效率提升** | 节省80%重复操作时间 | 标准化团队流程 | 提升整体生产力 |
| **质量保障** | 零人为错误 | 统一质量标准 | 建立最佳实践库 |
| **知识沉淀** | 个人技能资产化 | 团队知识共享 | 企业知识管理 |
| **成本优化** | 减少70% Token消耗 | 降低培训成本 | ROI可量化 |

---

**适用场景判断**：

**✅ 适合改造为Skill的场景**：
- 有明确的输入输出
- 流程可标准化（有明确步骤）
- 需要频繁重复（周/日级别）
- 有业务规则或判断逻辑
- 团队需要统一标准

**示例**：
- 📊 数据分析：周报、月报、营销数据
- 📝 内容创作：新闻稿、社交媒体、邮件模板
- 💻 研发流程：代码审查、需求澄清、部署检查
- 📋 业务流程：客户会议准备、合同审查、风险评估

---

**❌ 不适合改造为Skill的场景**：
- 一次性任务（不会重复）
- 高度个性化（每次都不同）
- 流程不明确（探索性工作）
- 简单任务（单步操作）

### 3.3 传统SOP到Skill的改造步骤【3分钟】

**6步改造法**：

```
Step 1: 识别工作流
├─ 列出完整的操作步骤
├─ 标注输入和输出
└─ 识别判断逻辑（if-else）

Step 2: 提取核心要素
├─ 必需的输入数据
├─ 关键的计算公式
├─ 重要的业务规则
└─ 期望的输出格式

Step 3: 标准化输入输出
├─ 定义输入数据格式（CSV、JSON等）
├─ 设计输出模板（表格、报告等）
└─ 明确字段要求

Step 4: 分层组织内容
├─ 元数据：name、description（触发关键词）
├─ 主指令：核心流程（保持简洁，< 500行）
└─ 参考资料：复杂规则（按需加载）

Step 5: 优化description触发
├─ 包含关键词（让AI能匹配到）
├─ 描述具体场景（避免泛泛而谈）
└─ 测试触发准确性

Step 6: 测试与迭代
├─ 正面测试：应该触发的场景
├─ 负面测试：不应该触发的场景
├─ 流程测试：完整执行一遍
└─ 收集反馈并优化
```
