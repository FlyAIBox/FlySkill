### 第8页：传统方式第二轮对话

**标题**：Step 2 - 补充效率指标

**Prompt示例**：
```markdown
效率分析 
为每个渠道计算以下额外指标： 

* ROAS（广告支出回报率）= 收入/支出
* CPA（单次获客成本）= 支出 / 转化次数
* 净利润 = 收入 - 总成本
  - 总成本 = 支出 + (订单数 × 运输成本) 
    + (收入 × 产品成本百分比)
  - 假设：平均单笔订单物流成本为 8 元，
    产品成本占比为 35%。
  
并与目标进行比较：
目标 ROAS：最低 4 倍  
最大 CPA：50 元
净利润：需为正值

输出格式：
请生成一个汇总表格，展示每个渠道的上述指标...
```

**问题标注**：
- 🔴 需要分多轮补充
- 🔴 成本参数需要每次输入
- 🔴 输出格式要求容易遗漏

### 第10页："小作文式Prompt"的三大痛点

**标题**：为什么这种方式不可持续？

**痛点1：重复劳动**
- 🔁 每次都要复制粘贴相同的长文本
- 📋 容易漏掉关键参数
- ⚙️ 无法版本管理和复用

**痛点2：质量不稳定**
- 👥 不同人写的Prompt效果差异大
- 📉 新员工需要长时间培训
- 🎯 很难确保每次都包含所有要点

**痛点3：知识无法沉淀**
- 📚 业务规则散落在个人的Prompt中
- 🔄 规则变更时需要全员同步更新
- 💼 员工离职带走宝贵的Prompt经验

**讲师注**：这三个痛点是推出Skill的核心动机，要讲透

### 第11页：更深层的技术问题

**标题**：Context Window的"拥挤"困境

**问题描述**：

**现象：Lost in the Middle**
- Context Window虽然在变大（已达200K tokens）
- 但拥挤的上下文会导致AI注意力分散
- 关键信息容易被"淹没"在中间部分

**示意图**：
```
[开头信息] → [大量中间信息] → [结尾信息]
   ✅           ❌ 容易被忽略      ✅
  记得清楚                      记得清楚
```

**后果**：
- 💰 Token消耗大，成本高
- ⚠️ 响应质量不稳定
- 🐌 推理速度变慢

**思考**：能否让AI"按需加载"知识，而不是一次性塞满？

**讲师注**：引出下一部分的解决方案 - 渐进式加载

### 第13页：什么是Agent Skill？

**标题**：从"长文本"到"文件系统标准"

**定义**：
> Agent Skill 是一种轻量级、开放的格式，用于扩展AI Agent的能力。  
> Skill是一个包含指令、脚本、资产和资源的**文件夹**，Agent可以通过发现这些Skill来准确执行特定任务。

**核心转变**：

| 传统方式 | Agent Skill |
|---------|-------------|
| 长文本Prompt | 文件夹 + 结构化文件 |
| 每次输入 | 一次安装，多次复用 |
| 个人维护 | 团队共享 + 版本控制 |
| 全量加载 | 按需加载 |

**关键特征**：
- 📦 **可移植**：可在Claude.ai、Claude Code、API间无缝使用
- 🧩 **可组合**：多个Skill可以组合完成复杂任务
- ⚡ **渐进式加载**：最小化Token消耗

**讲师注**：强调"文件夹"这个物理形态，让学员建立具象认知

### 第14页：核心理念 - 通用Agent + 可插拔Skill

**标题**：范式转移

**旧范式：专用Agent**
```
金融Agent          营销Agent          编程Agent
[脚手架+知识]     [脚手架+知识]     [脚手架+知识]
    ↓                ↓                ↓
知识隔离           维护困难          无法组合
```

**新范式：通用Agent + Skills**
```
         通用Agent（轻量脚手架）
              ↓ 按需加载
    [Skill库：可插拔的知识模块]
   /        |        |        \
金融Skill  营销Skill  编程Skill  法律Skill
   \        |        |        /
         可自由组合完成复杂任务
```

**优势对比**：
- ✅ 知识复用：一次创建，处处使用
- ✅ 降低维护成本：统一的脚手架
- ✅ 能力可组合：跨领域协作

**讲师注**：这是架构思维的转变，对技术同学要多讲一些

### 第15页：渐进式加载 - 三层架构

**标题**：如何解决Context Window拥挤问题？

**三层加载机制**：

| 层级 | 内容 | 何时加载 | Token消耗 |
|-----|------|---------|----------|
| **Level 1** | 元数据（name + description） | 系统启动时（始终） | 极低（~50 tokens/skill） |
| **Level 2** | 主指令（SKILL.md正文） | 触发时 | 中等（~2000 tokens） |
| **Level 3** | 参考资源（references/） | 需要时 | 按需（~1000-5000 tokens） |

**工作流程**：
```
1. Agent启动 → 加载所有Skill的元数据（Level 1）
   ↓ "我知道有100个技能可用"
   
2. 用户请求 → 匹配相关Skill → 加载主指令（Level 2）
   ↓ "这个任务需要营销分析Skill"
   
3. 执行过程 → 需要复杂规则 → 加载参考文档（Level 3）
   ↓ "预算调整需要详细规则，现在加载"
```

**对比传统方式**：
- 传统：一次性加载1500字 → 始终占用Context
- Skill：元数据50字 → 触发时才加载 → 用完可释放

**讲师注**：用具体数字让学员感受效率提升

### 第30页：好的description vs 坏的description

### 第33页：案例拆解 - 预算调整规则

**标题**：一个复杂references文件的设计

**文件**：`references/budget_reallocation_rules.md`（170行）

**内容组织**：
```markdown
# 预算调整规则

## 目录
* 规则 0：最低数据门槛
* 规则 1：渠道分类
* 规则 2：计算预算变更
* 多周调整逻辑
* 决策矩阵
* 要求的输出格式

## 规则 0：最低数据门槛
只有在分析周期内获得 >=50 次转化的渠道，
才具备调整预算的资格。

## 规则 1：渠道分类
将每个渠道归类为且仅归类为一个类别...

### 暂停 (PAUSE) -> 减少 100%
任意条件触发：
* 用户指出该渠道已连续 3 周以上表现负向/亏损
* 用户指出在经过 3 周以上的优化努力后仍无改善

### 大幅削减 (DECREASE_HEAVY) -> 减少 45%
...

## 决策矩阵
| 条件 | 分类 | 变更幅度 |
| --- | --- | --- |
| 负向表现 3 周以上 | 暂停 | -100% |
| ROAS < 50% 且 净利润 <= 0 | 大幅削减 | -45% |
...
```

**设计亮点**：
1. ✅ **目录**：让AI快速定位需要的规则
2. ✅ **层次清晰**：规则0、规则1、规则2
3. ✅ **决策矩阵**：用表格直观展示逻辑
4. ✅ **明确量化**：具体的百分比和阈值

**如果不用references会怎样？**
- ❌ SKILL.md会变成500+行的长文档
- ❌ 每次触发都加载170行的规则（即使不需要）
- ❌ 维护困难，规则修改要翻遍整个文档
