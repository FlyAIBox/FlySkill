# R&D 场景下的需求澄清测试提示词 (Test Prompts for Clarify Demand Uncertainty)

本文件包含一系列面向研发场景的测试提示词，旨在测试 Agent 是否能识别需求中的模糊性并提出关键的澄清问题。

## 1. 前端开发 (Frontend Development)

### 场景 A: 界面功能新增
**Prompt:**
> "在设置页面加一个夜间模式的开关。"

**为何模糊 (Ambiguity):**
- **位置:** 开关具体放在设置页面的哪个区域？
- **持久化:** 用户的选择是否需要保存到后端，还是仅保存在本地存储？
- **生效范围:** 切换后是立即应用到全局，还是仅当前页面？
- **系统跟随:** 是否需要支持“跟随系统设置”的选项？

### 场景 B: 现有组件修改
**Prompt:**
> "把首页的 Banner 图改得好看一点。"

**为何模糊 (Ambiguity):**
- **设计标准:** “好看”的主观定义是什么？有参考设计稿吗？
- **内容变更:** 图片内容或文案需要更换吗？
- **响应式:** 移动端和桌面端的展示逻辑是否需要调整？

## 2. 后端开发 (Backend Development)

### 场景 A: API 开发
**Prompt:**
> "写一个用户注册的接口。"

**为何模糊 (Ambiguity):**
- **字段:** 需要哪些必填字段（邮箱、手机号、用户名）？
- **验证:** 密码强度要求？是否需要邮箱/短信验证码验证？
- **安全:** 是否需要防刷限制（Rate Limiting）？
- **鉴权:** 注册后是否自动登录（返回 Token）？

### 场景 B: 性能优化
**Prompt:**
> "优化一下 `getUserList` 接口的性能。"

**为何模糊 (Ambiguity):**
- **目标:** 当前响应时间是多少？目标是多少？
- **手段:** 是优化 SQL 查询，还是加缓存？允许的数据一致性延迟是多少？
- **分页:** 是否需要调整分页逻辑？

## 3. 重构与维护 (Refactoring & Maintenance)

### 场景 A: 代码清理
**Prompt:**
> "清理一下 `utils.js` 文件里的代码。"

**为何模糊 (Ambiguity):**
- **范围:** 是删除未使用的函数，还是重写现有函数的实现？
- **风险:** 是否允许破坏性变更（Breaking Changes）？
- **规范:** 有指定的代码风格指南吗？

### 场景 B: 技术栈升级
**Prompt:**
> "把项目升级到 React 19。"

**为何模糊 (Ambiguity):**
- **兼容性:** 如何处理废弃的 API？
- **测试:** 需要多大范围的回归测试？
- **时间线:** 是一次性全部升级，还是渐进式升级？

## 4. 运维与基础设施 (DevOps & Infrastructure)

### 场景 A: 流水线搭建
**Prompt:**
> "给项目配一套 CI/CD。"

**为何模糊 (Ambiguity):**
- **工具:** 使用 GitHub Actions, Jenkins 还是 GitLab CI？
- **流程:** 包含哪些阶段（Lint, Test, Build, Deploy）？
- **环境:** 部署到测试环境还是生产环境？

### 场景 B: 部署
**Prompt:**
> "把应用部署到 AWS 上。"

**为何模糊 (Ambiguity):**
- **服务:** 使用 EC2, Lambda, 还是 App Runner？
- **配置:** 需要什么样的机器配置？
- **网络:** VPC 和安全组规则有什么要求？

## 5. Bug 修复 (Bug Fixing)

### 场景 A: 报错修复
**Prompt:**
> "修复一下登录报错的问题。"

**为何模糊 (Ambiguity):**
- **现象:** 具体报什么错（500, 401, 还是前端 JS 报错）？
- **复现:** 在什么环境下出现的？复现步骤是什么？
- **账号:** 是所有用户都不行，还是特定账号不行？

---
**使用说明:**
使用此 Skill 时，Agent 应针对上述 Prompt 提出 1-5 个多项选择题，帮助用户明确需求范围、验收标准和潜在约束。
